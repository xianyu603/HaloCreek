# 公共服务聚合

本文记录应用级公共服务聚合的边界。它解决的是“很多 service / ViewModel 都合理需要使用同一批基础设施能力”的问题，不解决当前 workspace、配置派生或业务状态管理问题。

## 定位

公共服务聚合可以理解为应用运行环境中的基础设施入口。它提供的是平台能力和统一错误处理政策，而不是某个业务域的状态。

命名：

- `AppCommonRuntime`

当前适合纳入的对象：

- `PlatformInfrastructure`
- `ApplicationStatusService`
- `TransientEventService`

这些对象的共同点是：任何 service 或 ViewModel 都可以理直气壮地依赖它们，因为它们表达的是“在 HaloCreek 这个桌面应用里如何执行平台公共操作、如何报告用户可见错误、如何维护全局状态展示策略”。

## 应该收口的能力

- 打开目录选择器、错误弹窗、终端窗口等平台公共操作。
- 一过性用户动作失败的统一报告入口。
- 持续性全局状态和后台任务状态的统一入口。
- 后续其他应用级策略能力，例如应用级诊断、统一用户通知策略。

## 不应该收口的能力

- 当前 workspace。
- 当前 workspace 派生出的 effective config。
- Git、tmux、session history、session lifecycle 等业务能力服务。
- 任何只是为了少传一个参数而塞进来的业务状态。

公共服务聚合不应该变成 service locator。它只适合承载横切基础设施，不适合成为“所有服务都从这里拿”的万能对象。

## 依赖原则

业务 service 可以依赖公共服务聚合，只要依赖的是平台能力或错误处理政策。例如某个 service 在执行用户动作时需要按统一策略弹窗并写日志，可以依赖公共错误处理入口。

公共服务聚合不应该依赖具体业务 service。否则它会从基础设施入口变成应用总管，边界会变模糊。

公共服务聚合是否依赖应用状态/配置聚合不是原则问题。如果错误处理政策确实需要根据当前 workspace 或 effective config 改变，公共服务聚合依赖它是可以接受的。

需要提前想清楚的是两个工程风险：

- 启动和配置加载失败时，错误处理入口本身仍然要可用。否则“配置坏了”可能导致“报告配置坏了的能力也坏了”。
- 如果应用状态/配置聚合在选择 workspace、校验 workspace、加载配置失败时又需要调用公共错误处理入口，双方直接依赖会形成循环依赖。这个循环可以通过应用组装层编排、事件回调或更窄的策略接口拆开。

目前暂时没有公共处理政策走配置的需求，推迟到有实际需求时思考这两个可能的工程风险。

## 与应用状态/配置聚合的关系

公共服务聚合和应用状态/配置聚合是两件独立的事，概念边界独立，但实现上可以存在依赖关系。

两者可以在应用组装层同时创建，并被不同对象按需依赖：

- service / ViewModel 需要平台和错误策略时依赖公共服务聚合。
- ViewModel / 编排层需要当前 workspace 或 effective config 时依赖应用状态/配置聚合。

如果某个 workspace 切换流程需要报告错误，推荐由调用流程同时使用两者，而不是让其中一个服务强依赖另一个服务。

如果某条公共错误处理政策确实来自配置，可以让公共服务聚合直接读取应用状态/配置聚合，也可以把该政策抽成更窄的策略读取接口。选择哪种方式主要看是否会造成初始化顺序、失败兜底或循环依赖问题。

## 实施约束

- 只收横切基础设施，不收业务 service。
- 依赖应显式出现在构造函数中，不通过静态全局对象隐式读取。
- 不以减少构造参数为唯一目标；只有当能力本身确实是应用级公共能力时才纳入。
- 命名避免使用 `AppContext`，以免和 `System.AppContext` 混淆。
