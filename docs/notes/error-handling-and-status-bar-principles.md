# Error Handling And Status Bar Principles

本文记录 HaloCreek 当前阶段对错误处理和 status bar 的使用原则。

## 设计目标

Status bar 不再作为一过性事件的提示板，而是只表达用户当前仍需要知道的全局状态。

当前阶段优先保持 happy path 清晰、界面克制、实现简单。Tab-local 错误展示暂不系统性建设，后续根据真实使用中反复出现的用户困惑再选择性补充。

## Status Bar 使用原则

Status bar 只显示两类状态：

1. 正在进行的后台任务。
2. 持续性的全局错误状态。

持续性全局错误指应用当前仍处于不可用或关键能力不可用的状态，例如配置损坏、关键依赖不可用、应用启动后无法建立必要上下文等。

Status bar 不显示一过性事件，包括：

- 普通操作成功。
- 用户取消操作。
- 后台刷新成功。
- 可恢复的后台刷新失败。
- 外部动作已经尝试启动。
- 调试性诊断信息。

这些事件默认只写日志，方便调试，不打扰用户，也不抢占全局状态位置。

## 错误处理原则

错误按用户影响分层处理。

### 一过性错误

一过性错误是某个操作或某次后台任务失败，但失败状态不会持续定义应用当前状态。

如果错误来自用户刚刚触发的动作，并且用户必须知道结果，使用弹窗提示。提示中应说明失败原因和已采取的兜底行为。

如果设计上允许兜底，必须显式处理，不做静默降级。用户可感知的降级需要弹窗或后续可见反馈；用户不可感知且不影响判断的失败可以只写日志。

如果错误来自后台任务，并且现有数据或界面仍可继续使用，默认只写日志。不要为了偶发后台失败打断用户，也不要把 status bar 变成后台诊断流。

### 持续性错误

持续性错误是当前应用状态仍然错误，并且会影响全局可用性。

这类错误应持续显示在 status bar，直到状态恢复或用户完成必要处理。持续性错误优先级高于普通后台任务，不应被后台任务提示覆盖。

程序 bug 或违反设计前置条件的状态也按持续性错误处理。例如 workspace 为空已经从设计上禁止，如果仍出现，应作为应用不可用状态暴露，而不是假装继续运行。

## Tab-local 错误

Tab 内错误展示当前阶段不系统性建设。

默认策略是：

- Tab 内优先保证 happy path。
- 必要空态可以保留。
- 非阻断型异常先日志化。
- 不为每个 tab 立即引入完整的错误态、旧数据态、重试入口和局部优先级模型。

History 当前的列表处理是基准：优先显示可用列表；后台刷新失败时保留旧结果，不急着把 tab 变成错误页。

后续只有当某类 tab-local 错误反复造成用户困惑，才为该具体场景补充局部展示。

## 实现约束

- 不把 status bar 当作任意 ViewModel 的通用消息输出口。
- 不让一过性消息覆盖持续性错误。
- 不为了调试信息抢占用户可见状态。
- 允许降级时必须明确降级边界和用户可感知影响。
- 无法安全降级时应失败、弹窗或暴露持续性错误，不静默吞掉。
