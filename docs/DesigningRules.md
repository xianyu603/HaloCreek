# Designing Rules

0. 编号更小的规则优先级更高。已有代码和 Designing Rules 冲突时，新代码以 Designing Rules 为准，修改既有代码时也优先向 Designing Rules 收束。
1. 可以用非核心行为的适当降级换取代码明显精简，记录降级内容和用户可感知影响即可。
2. 模块应保持明确边界。类/方法/参数应语义明确，避免必须阅读实现才知道其定位、输入输出或行为边界。
3. 禁止猜接口或静默降级；外部协议、文件格式、API 字段取不到就留空、失败或显式报告。
4. 短且线性的单点调用不要拆成很多私有方法，禁止很薄的转发方法。
5. 错误处理默认抛异常，不用 Result/OperationResult 包装失败；只有返回值本身就是业务数据，且成功/失败都属于正常业务语义时，才允许使用明确命名的结果对象。
6. 不为尚不存在的生命周期、重入、并发或扩展场景提前加防御代码；对象按当前真实生命周期设计，只有已有调用路径确实需要时，才引入 Dispose 状态、锁、取消、版本判断或复杂接口。
