# Skill 选择体验方案

## 目标

Skill 补全用于在 prompt 输入框中快速显式引用 Codex skill，降低记忆具体 skill 名称和判断 skill 来源的成本。

首版重点解决两个用户体验问题：

- 忘记 skill 名称时，可以通过来源和分类逐层浏览。
- 看到具体 skill 时，可以判断它大致来自哪里，而不是只看到一个孤立名称。

本方案只定义选择体验和展示语义。Skill 发现、解析、约束行为和具体实现细节先不展开。

## 触发入口

Skill 使用 `$` 作为触发字符。

用户在 prompt 输入框中输入满足自动补全触发规则的 `$token` 后，候选框展示 skill 候选。`$` 之后的文本称为 query。

Skill 补全复用自动补全框架的既有键盘行为：

- `Down` / `Up` 在当前列表中移动选择。
- `Right` 在当前项有下级菜单时进入下级菜单。
- `Left` 从下级菜单返回上一级。
- `Enter` 接受当前选中项并写回输入框。
- `Esc` 退出当前选择态。

接受具体 skill 后，补全框架按既有规则替换当前 `$token`，并在插入文本后追加空格。

## 空 Query 的导航结构

当用户只输入 `$`，即 query 为空时，候选框进入浏览模式。

浏览模式采用三级结构：

```text
来源 -> 分类 -> skill
```

### 一级：来源

来源是显式分组，用于回答“这个 skill 从哪里来”。

首版来源分组按可识别的信息展示，建议包括：

- `Project`：当前项目或 workspace 可见的 skill。
- `User`：用户个人安装或配置的 skill。
- `Admin`：机器或组织级别提供的 skill。
- `System`：Codex 内置 skill。
- `Plugin`：插件提供的 skill；如果能识别具体插件，可在描述中显示插件名。

来源分组不依赖启发式分类。无法可靠识别来源的 skill 不应静默混入其他来源，可以进入 `Other` 或等价的兜底来源。

### 二级：分类

分类是浏览辅助，用于在一个来源下减少列表长度。

分类可以启发式判断，不要求绝对准确。首版建议保持少量稳定分类：

- `信息获取`：文档查询、API 查询、搜索、读取历史或只读上下文类 skill。
- `代码编写`：实现、修改、重构、代码审查、框架相关编码类 skill。
- `动作执行`：编译、测试、启动、安装、发布、生成资产、调用外部工具类 skill。
- `管理配置`：skill/plugin 创建、安装、配置、治理类 skill。
- `其他`：无法稳定归类的 skill。

分类只影响浏览路径，不影响 skill 是否可用，也不表达权限或安全级别。

### 三级：具体 Skill

具体 skill 是唯一可接受并写回 prompt 的条目。

展示建议：

- 标题显示 skill 名称。
- 说明优先显示来源、短路径或插件名，再显示 skill 描述。
- 同名 skill 不合并，全部展示。

示例：

```text
Project
  信息获取
    avalonia-api-docs
      Project · .codex/skills/avalonia-api-docs · Find version-matched Avalonia API documentation...

System
  管理配置
    skill-creator
      System · bundled · Create or update a skill...
```

## 有 Query 时的搜索结构

当 query 非空时，候选框进入快速定位模式。

快速定位模式不强制用户继续走 `来源 -> 分类 -> skill` 的三级路径。用户已经输入 query 时，优先把可接受的具体 skill 推到当前列表中。

匹配优先级：

1. 精确匹配来源别名。
2. 精确匹配分类别名。
3. 精确匹配 skill 名称。
4. 全部 skill 模糊匹配。

### 精确匹配来源

当 query 精确匹配某个来源名称或来源别名时，候选框直接展示该来源下的分类列表。

示例：

```text
输入: $project
一级列表:
- 信息获取
- 代码编写
- 动作执行
- 管理配置
- 其他
```

### 精确匹配分类

当 query 精确匹配某个分类名称或分类别名时，候选框直接展示所有来源下该分类内的具体 skill。

每个 skill 的说明中仍然显示来源，避免跨来源搜索结果失去 provenance。

示例：

```text
输入: $code
一级列表:
- avalonia-api-docs
  Project · 信息获取 · .codex/skills/avalonia-api-docs
- skill-creator
  System · 管理配置 · bundled
```

如果分类别名和来源别名冲突，来源别名优先。

### 精确匹配 Skill 名称

当 query 精确匹配一个或多个 skill 名称时，匹配到的具体 skill 展示在列表前部。

如果存在同名 skill，全部展示，并通过说明区分来源和路径。接受任一同名项时，首版仍写回相同的 `$skill-name`。

示例：

```text
输入: $build-review
一级列表:
- build-review
  Project · 动作执行 · .codex/skills/build-review
- build-review
  User · 动作执行 · ~/.codex/skills/build-review
```

### 全部 Skill 模糊匹配

当 query 未精确匹配来源、分类或 skill 名称时，在全部具体 skill 中做模糊匹配，并把匹配到的具体 skill 作为一级列表展示。

模糊匹配结果不再保留多级菜单。每个结果通过说明展示来源、分类和短路径。

首版可匹配：

- skill 名称。
- skill 描述。
- 来源名称。
- 分类名称。
- 路径中的关键片段。

匹配大小写不敏感。

## 排序规则

首版排序以稳定、可解释为主，不引入最近使用、权重学习或语义排序。

建议顺序：

- 空 query 时，来源按固定顺序展示，分类按固定顺序展示，skill 在各分类内按名称或发现顺序稳定展示。
- 精确匹配来源时，按该来源下的固定分类顺序展示。
- 精确匹配分类时，按来源固定顺序分组展开，再展示具体 skill。
- 精确匹配 skill 名称时，精确匹配项排在模糊匹配项之前。
- 模糊匹配时，优先展示名称命中的 skill，再展示描述、来源、分类或路径命中的 skill。

首版不做拼音排序、分词排序、语义相似度排序和最近使用排序。

## 同名 Skill

同名 skill 不合并。

展示规则：

- 标题保持原始 skill 名称。
- 说明中必须包含来源或短路径，用于区分同名项。

写回规则：

- 首版仍写回普通 `$skill-name`。
- 不发明 `$source/skill`、`$plugin:skill` 等新语法，除非确认 Codex 支持。

同名项的主要价值是让用户知道当前环境存在冲突，而不是在 HaloCreek 内自行定义新的消歧协议。

## 候选项模型边界

首版优先复用现有补全候选项能力。

来源、分类、路径和重复提示可以先压缩进标题或说明文本中展示。只有在后续出现以下需求时，再考虑扩展候选项模型：

- UI 需要独立 badge、图标或固定来源列。
- 需要打开 skill 文件、查看详情、禁用 skill 或执行管理动作。
- 需要在接受候选时携带结构化来源信息，而不只是写回 `$skill-name`。

在只有“选择并插入 skill 引用”的阶段，不为了预期能力提前扩展模型。

## 空状态

当 query 非空且没有任何来源、分类或 skill 匹配时，候选框按自动补全框架的既有空状态展示。

Skill 补全不自动创建 skill，不自动跳转安装流程，也不弹出额外说明窗口。

## 非目标

首版不做：

- Skill 发现路径和扫描策略的详细实现。
- Skill frontmatter 解析细节。
- Skill 约束行为。
- Skill 详情页。
- Skill 安装、卸载、启用、禁用。
- 打开 skill 文件或跳转到来源目录。
- 同名 skill 的新引用语法。
- 最近使用排序。
- 语义搜索。
- 分类编辑 UI。
- 来源筛选配置。

这些能力在 skill 选择体验稳定后再单独设计。
