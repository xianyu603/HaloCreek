## 目标

在现有 `PromptInputViewModel` 和 `PromptInputView` 上先实现一条最小可用的自动补全交互链路：

1. 用户在 prompt 输入框中输入 `$` 后，根据光标前文本判断是否显示补全菜单。
2. 菜单显示后位置固定，不随继续输入时的文本布局变化跳动。
3. 用户可以用方向键进入和导航菜单。
4. 用户按回车后，将当前选中的补全文本写回输入框。
5. 后续快捷语、skill 快捷引用、快捷文件输入都复用同一套菜单状态和键盘交互。

首版触发字符写死为 `$`，不接配置，不接真实 skill / 文件数据源。菜单项先使用固定数据，优先验证 UI 行为和输入框写回路径。

## 触发规则

输入框每次文本或光标位置变化时，检查光标前的当前 token。

显示菜单的条件：

- 光标前存在最近的 `$`。
- 从这个 `$` 到光标之间没有空格。
- `$` 位于文本开头，或 `$` 前一个字符是允许的分隔字符。
- 光标仍在该 token 内，不允许跨过空格后继续保持菜单。

首版允许的分隔字符固定为：

- 空白字符：空格、Tab、换行。
- 中文标点：`，`、`。`、`；`、`：`、`（`、`【`、`《`、`“`、`‘`。
- 英文标点：`,`、`.`、`;`、`:`、`(`、`[`、`{`、`"`、`'`。

首版只认上面的白名单，不根据文件路径、变量名或 Markdown 语义额外放宽。比如 `abc$` 不触发，`/tmp/$` 不触发，`foo_bar$` 不触发。

隐藏菜单的条件：

- 光标前不存在满足条件的 `$`。
- `$` 到光标之间出现空格。
- 光标移动到触发 token 外。
- 用户回车接受菜单项后。

首版不做复杂解析，不识别引号、转义、Markdown 语义或文件路径语义。判断逻辑只服务当前 prompt 输入框的 `$token` 场景。

## 菜单位置

菜单第一次显示时，根据当前光标位置计算弹出点。

菜单显示后：

- 弹出点保持固定。
- 用户继续输入过滤文本时，菜单不跟随光标移动。
- 只有菜单隐藏后再次满足触发条件，才重新计算弹出点。

这样可以避免多行输入框中弹层随换行、滚动或字体测量产生明显跳动。

## UI 样式

补全菜单是输入框的局部弹层，不是全局窗口。

基础样式：

- 宽度固定或设置最小宽度，避免条目文字变化导致菜单横向抖动。
- 单项高度固定，支持图标预留位、主标题和可选说明文本。
- 当前选中项使用明确的背景色和左侧强调线。
- 空状态显示置灰的一个Empty条目
- 允许多级菜单，多级菜单从当前项右侧弹出。

菜单不承载解释性帮助文案。用户应该通过条目标题、说明和键盘反馈理解当前可执行动作。

## 键盘行为

输入框仍然是键盘事件入口。菜单显示时，部分按键由自动补全逻辑优先处理。

基础行为：

- `Down`：如果菜单显示且还未进入菜单，选中第一项；如果已有选中项，移动到下一项。
- `Up`：移动到上一项。
- `Right`：当前项有子菜单时打开子菜单，并选中子菜单第一项。
- `Left`：当前处于子菜单时返回父菜单。
- `Enter`：接受当前选中项，将补全文本写入输入框。
- `Esc`：离开菜单，菜单返回为初始且未选中的样式。

与现有快捷键的关系：

- `Ctrl+Enter` 继续执行 `LaunchCommand`。
- `Alt+Enter` 继续执行 `SendToFrontCommand`。
- 菜单显示时，普通 `Enter` 优先用于接受补全项；没有选中项时不接受。

方向键被菜单消费后，不再传给输入框移动光标。菜单隐藏时，方向键保持 TextBox 默认行为。

## 文本写回规则

接受补全项时，只替换触发 token 所在的完整片段，不替换整段 prompt。

替换区间从触发字符 `$` 开始，到 `$` 之后遇到的下一个空白字符之前结束。首版空白字符按空格、Tab、换行处理。如果光标停在 token 中间，也仍然替换到 token 结束。

示例：

```text
输入前: 先口头汇报 $ski
选中项: $skill:code-review
输入后: 先口头汇报 $skill:code-review 

输入前: $1234 abc
光标位置: 3 后面
选中项: $skill:code-review
输入后: $skill:code-review abc
```

写回步骤：

1. 保存触发 token 的起始位置、结束位置和触发字符。
2. 用户回车接受时，从 `start` 向后查找下一个空白字符，得到实际替换结束位置 `replaceEnd`。
3. 用选中项的插入文本替换 `[start, replaceEnd)` 区间。
4. 如果 `replaceEnd` 已经是原文本末尾，则在插入文本后自动追加一个空格。
5. 将光标移动到插入文本末尾；如果自动追加了空格，则移动到空格之后。
6. 关闭菜单。
7. 触发 `PromptText` 更新，让现有命令可执行状态继续刷新。

如果替换区间后面已经有其他字符，首版不额外插入空格，保留原文本中的分隔。

## ViewModel 状态

`PromptInputViewModel` 增加自动补全相关页面状态，但不引入独立服务。

建议状态：

- `IsCompletionOpen`：菜单是否显示。
- `CompletionAnchor`：菜单固定弹出点，具体类型按 Avalonia 绑定需要确定。
- `CompletionItems`：当前一级菜单项。
- `SelectedCompletionPath`：当前选中路径，用于支持多级菜单。
- `CompletionTokenStart` / `CompletionTokenEnd`：当前触发 token 的文本区间。
- `CompletionFilterText`：`$` 后到光标前的过滤文本。

这些状态属于 prompt 输入组件自身的交互状态，不应放到 `ReviewViewModel` 或 `PromptEditorViewModel`。两个页面继续只持有同一个 `PromptInputViewModel`。

首版菜单项可以定义为简单模型，例如：

```csharp
public sealed class PromptCompletionItem
{
    public required string Title { get; init; }
    public string? Description { get; init; }
    public required string InsertText { get; init; }
    public IReadOnlyList<PromptCompletionItem> Children { get; init; } = Array.Empty<PromptCompletionItem>();
}
```

如果实际实现中 Avalonia 绑定要求可观察集合，可以在 ViewModel 内部使用 `ObservableCollection`，但对外语义仍然是“当前菜单项集合”。

## View 接入

`PromptInputView.axaml` 继续以 `TextBox` 为主体，在同一个 `UserControl` 内增加补全弹层。

View 侧职责：

- 监听 TextBox 的光标位置变化和按键事件。
- 把文本、光标位置、按键意图传给 ViewModel。
- 根据 ViewModel 状态显示或隐藏菜单。
- 在需要时把菜单接受后的光标位置设置回 TextBox。

ViewModel 侧职责：

- 判断触发 token 是否有效。
- 维护菜单打开、关闭、选中项和过滤文本。
- 决定接受项后应生成的新文本和新光标位置。
- 打印显示 / 隐藏日志。

不把 Avalonia 控件对象传入 ViewModel。需要 caret rect、TextBox selection 等 UI 细节时，由 View 读取后传入必要的值。

## 日志

step1 必须先能通过日志验证显示 / 隐藏判断。

日志分类沿用 `PromptInput`。

建议日志：

- 菜单显示：记录 token 起点、光标位置、过滤文本。
- 菜单隐藏：记录隐藏原因，例如 `NoTrigger`、`WhitespaceAfterTrigger`、`CaretOutsideToken`、`Escape`、`Accepted`。
- 接受菜单项：记录插入项标题和替换区间。

日志不打印完整 prompt 文本，避免把用户输入内容写入日志。

## 实施步骤

### step1 内容检查和显示状态

修改 `PromptInputViewModel` 和 `PromptInputView`：

1. 在 TextBox 文本变化或光标变化时调用触发检查。
2. 写死触发字符 `$`。
3. 判断是否显示 / 隐藏菜单。
4. 显示 / 隐藏时打印日志。
5. UI 可以先只显示一个简单弹层或占位菜单。

验收：

- 输入 `$` 时菜单显示。
- 输入 `abc$` 不显示；输入 `abc $`、`abc，$`、`abc($` 显示。
- 输入 `$abc` 后继续输入仍保持显示。
- 输入 `$abc def` 后菜单隐藏。
- 光标移动到 token 外后菜单隐藏。

### step2 单级菜单导航

在固定测试数据上实现一级菜单：

1. 菜单显示时加载固定条目。
2. `Down` / `Up` 可以移动选中项。
3. 方向键被菜单消费，不移动 TextBox 光标。
4. `Esc` 关闭菜单。

验收：

- 第一次按 `Down` 选中第一项。
- 继续按 `Down` / `Up` 可以循环或边界停留，具体策略固定即可。
- 菜单关闭后方向键恢复输入框默认行为。

### step3 菜单回车输入

实现接受当前菜单项：

1. `Enter` 接受当前选中项。
2. 替换触发 token。
3. 更新 TextBox 光标位置。
4. 关闭菜单并打印接受日志。

验收：

- `$ski` 接受后替换 `$ski`，并在末尾自动追加一个空格。
- `$1234 abc` 且光标在 `3` 后面时，接受后替换完整 `$1234`，保留后面的 ` abc`。
- prompt 中 `$` 前后的其他文本不变。
- `Ctrl+Enter` 和 `Alt+Enter` 不受影响。

### step4 多级菜单导航逻辑

在固定测试数据中加入子项：

1. `Right` 打开子菜单。
2. `Left` 返回父菜单。
3. `Up` / `Down` 只在当前层级内移动。
4. `Enter` 接受当前层级选中项；如果父项不可插入且只有子项，则不接受或进入子菜单，策略需固定。

验收：

- 一级和二级选中状态清晰。
- 子菜单打开后父菜单位置不变。
- 返回父菜单后仍保留父级选中项。

## 非目标

首版不实现：

- 配置化触发字符。
- 真实 skill 列表加载。
- 文件路径或 git 修改文件补全。
- 模糊搜索排序。
- 异步数据源和取消。
- 多窗口共享补全状态。
- 粘贴图片、结构化 prompt 模板、右键 action。

这些能力等 UI 行为稳定后再接入同一套菜单模型。

## 风险和降级

- Avalonia TextBox caret 精确坐标如果难以稳定获取，菜单位置可以先降级为输入框固定位置。
- 多行输入框滚动时，固定菜单可能与当前光标视觉距离变远；首版接受该行为，因为 draft 明确要求菜单弹出后位置保持固定。
- 多级菜单的鼠标悬停行为可以延后，首版只保证键盘路径。
- 如果 TextBox 的 KeyBinding 先于 KeyDown 消费 `Enter`，需要在 View 层明确事件处理顺序，确保普通 `Enter`、`Ctrl+Enter`、`Alt+Enter` 的边界可测试。
