# MVP1-6 T03-T08：Git File Browser 配置驱动文件动作实施方案

## 1. 目标

本方案替代原 `MVP1-6-git-file-browser-MVP实施方案.md` 中 T03 到 T08 的固定按钮实现口径。

新的目标是：Git File Browser 的工具栏按钮数量、按钮排布、按钮执行行为和双击行为都由配置决定。HaloCreek 只负责把当前 workspace、选中的 Git 变更文件和配置拼成一次外部进程启动请求；进程能否表达正确业务语义、是否真的完成 diff/open/edit/commit，由用户配置和外部工具决定。

默认配置先满足当前希望的行为：

- `OpenDiff`：按钮动作，调用 `TortoiseGitProc.exe` 的 Diff。
- `Open`：双击动作，用默认应用打开选中文件，不默认占一个按钮。
- `ExploreTo`：按钮动作，调用 `explorer.exe` 并选中选中文件。
- `Edit`：按钮动作，调用 `notepad.exe` 打开选中文件。
- `Commit`：按钮动作，调用 `TortoiseGitProc.exe` 的 Commit，在 workspace 根目录执行；UI 上作为右侧/root action 分组展示。

## 2. 当前基础

已经完成或基本可复用的部分：

- `GitService.GetChanges(...)` 已接入 `git status --porcelain=v1 -z`，能返回 `GitChangesResult`。
- `GitViewModel` 已有 `WorkspacePath`、`Changes`、`SelectedChange`、`RefreshCommand` 和 Footer 状态分发。
- `GitView.axaml` 已有列表、空态和固定按钮骨架，但按钮仍是写死且禁用。
- `AppConfig` / `ConfigService` 已有分层 JSON 配置读取能力，可以扩展配置字段。

本方案继续保持 ViewModel 不直接调用 `Process.Start`，UI 不直接解析 Git 输出。

## 3. 核心设计

### 3.1 配置模型

在 `AppConfig` 中新增 Git File Browser 动作配置：

```json
{
  "GitFileBrowserDoubleClickActionId": "Open",
  "GitFileBrowserActions": [
    {
      "Id": "OpenDiff",
      "Label": "OpenDiff",
      "ShowAsButton": true,
      "Placement": "Left",
      "Executable": "TortoiseGitProc.exe",
      "Arguments": ["/command:diff", "/path:{SelectedPath}"]
    },
    {
      "Id": "Open",
      "Label": "Open",
      "ShowAsButton": false,
      "Placement": "Left",
      "Executable": "explorer.exe",
      "Arguments": ["{SelectedPath}"]
    },
    {
      "Id": "ExploreTo",
      "Label": "ExploreTo",
      "ShowAsButton": true,
      "Placement": "Left",
      "Executable": "explorer.exe",
      "Arguments": ["/select,{SelectedPath}"]
    },
    {
      "Id": "Edit",
      "Label": "Edit",
      "ShowAsButton": true,
      "Placement": "Left",
      "Executable": "notepad.exe",
      "Arguments": ["{SelectedPath}"]
    },
    {
      "Id": "Commit",
      "Label": "Commit",
      "ShowAsButton": true,
      "Placement": "Right",
      "Executable": "TortoiseGitProc.exe",
      "Arguments": ["/command:commit", "/path:{WorkspaceRoot}"]
    }
  ]
}
```

建议新增模型：

- `GitFileBrowserActionConfig`
- `GitFileBrowserActionPlacement`：`Left` / `Right`

字段语义：

- `Id`：动作稳定标识，双击动作通过它引用。
- `Label`：按钮文本。
- `ShowAsButton`：是否显示为工具栏按钮；设为 `false` 的动作仍可被双击引用。
- `Placement`：按钮排布分组。`Commit` 默认放到右侧，让 root action 和文件 action 视觉上分开。
- `Executable`：要启动的程序，例如 `TortoiseGitProc.exe`、`explorer.exe`、`notepad.exe`。本字段不做 token 替换。
- `Arguments`：参数列表，每个元素独立替换 token 后加入 `ProcessStartInfo.ArgumentList`。

不把 `WorkingDirectory` 暴露成配置字段。动作统一使用 `{WorkspaceRoot}` 作为工作目录。

不把 `UseShellExecute` 暴露成配置字段。所有动作都走同一套普通进程启动逻辑。默认 `Open` 通过 `explorer.exe "{SelectedPath}"` 交给 Windows 默认打开行为，因此不需要把 `{SelectedPath}` 写进 `Executable`，也不需要为 `Open` 增加专用执行分支。

配置合并策略保持简单：全局配置先给默认值，workspace `.HaloCreek/config.json` 如果提供 `GitFileBrowserActions`，就整体替换动作列表，不做按 `Id` 的深度合并。这样按钮数量和行为完全由最终配置决定。

### 3.2 Token

动作执行时只做必要 token 替换：

- `{WorkspaceRoot}`：当前 workspace 路径。当前阶段把用户选择的 workspace 视为项目根目录。
- `{SelectedRelativePath}`：选中变更的 `RelativePath`。
- `{SelectedPath}`：`Path.Combine(WorkspaceRoot, SelectedRelativePath)` 得到的绝对路径。
- `{SelectedOriginalRelativePath}`：rename/copy 原路径；没有则为空字符串。
- `{SelectedOriginalPath}`：原路径对应绝对路径；没有则为空字符串。
- `{SelectedDisplayPath}`：`GitChangeInfo.DisplayPath`。

不做复杂模板语言，不支持条件、循环或环境变量展开。

### 3.3 执行语义

新增一个通用动作执行入口，先放在 `GitService` 中，或新增平铺服务 `GitFileActionService`：

```csharp
GitOperationResult TryRunConfiguredAction(
    string? workspacePath,
    GitChangeInfo? selectedChange,
    GitFileBrowserActionConfig action);
```

执行规则：

- 所有动作都要求有 workspace。
- 是否要求选中文件由 token 推断：`Arguments` 中出现 `{Selected...}` token 时要求有 `SelectedChange`；只使用 `{WorkspaceRoot}` 的动作不要求选中文件。
- 不根据 `GitChangeType` 做额外禁用，不特别拦截 deleted/untracked/conflicted。外部工具能不能处理，由配置和工具自己决定。
- 不预先判断目标文件是否存在。比如 deleted 文件被 `Open` 或 `Edit` 打开失败，直接把进程启动失败或外部工具行为交给用户配置。
- `Process.Start` 没抛异常就视为 HaloCreek 层面的成功，Footer 显示类似 `Started OpenDiff.`。
- 进程启动抛出异常、缺少 `Executable`、缺少必须 token 对应的数据时，Footer 显示失败原因。
- 不等待进程退出，不根据退出码判断业务成功，不在动作后自动刷新 Git 列表。

这个语义有意保持薄：HaloCreek 只负责“尝试执行一次配置的动作”。

## 4. UI 和 ViewModel 调整

### 4.1 GitViewModel

建议新增：

- `IReadOnlyList<GitFileActionButtonViewModel> LeftActionButtons`
- `IReadOnlyList<GitFileActionButtonViewModel> RightActionButtons`
- `IRelayCommand<GitFileActionButtonViewModel> RunActionCommand`
- `IRelayCommand<GitChangeInfo> OpenSelectedChangeCommand` 或 `DoubleClickCommand`

行为：

- workspace 切换或刷新时读取有效配置，生成按钮集合。
- `ShowAsButton=false` 的动作不进按钮集合，但仍可作为双击动作。
- 按钮 `CanExecute` 只按 token 做最小判断：有 workspace 即可；如果 `Arguments` 中出现 `{Selected...}` token，则还要求有选中项。
- `SelectedChange` 变化时刷新按钮 `CanExecute`。
- 双击列表行时，读取 `GitFileBrowserDoubleClickActionId` 对应动作并执行；找不到配置则 Footer 报告明确状态。

### 4.2 GitView.axaml

工具栏由固定按钮改为绑定集合：

- 左侧 `ItemsControl` 绑定 `LeftActionButtons`。
- 右侧新增 root/action 分组 `ItemsControl` 绑定 `RightActionButtons`。
- `Refresh` 仍在最右侧，和右侧动作保持视觉间距。

推荐布局：

```text
[OpenDiff] [ExploreTo] [Edit]                         [Commit] [Refresh]
```

`Open` 默认不展示按钮，因为它绑定到双击行为。用户如果希望同时显示按钮，只要把 `Open.ShowAsButton` 配成 `true`。

列表行接入双击事件即可，不改变现有三列表格展示。

## 5. T03-T08 新任务拆分

- [ ] **6-T03 UI 接入配置**：扩展 `AppConfig` / `ConfigService`，新增 Git File Browser 动作配置模型、默认动作列表和双击 action id；把 `GitView.axaml` 中固定禁用按钮改为左右两组动态按钮，`GitViewModel` 根据有效配置生成按钮集合并接入双击动作选择。验收点是无用户配置时能得到默认 `OpenDiff`、`Open`、`ExploreTo`、`Edit`、`Commit` 五个动作；默认工具栏显示 `OpenDiff`、`ExploreTo`、`Edit` 在左侧，`Commit` 在右侧，`Open` 不显示按钮但被双击引用；修改配置后按钮数量、文本、左右排布和双击 action id 都随配置变化。
- [ ] **6-T04 行为接入配置**：实现 `Executable + Arguments` 单一动作执行模型、token 替换、最小 `CanExecute` 判断和统一 `RunActionCommand`。验收点是按钮点击和列表双击都只尝试执行对应配置动作；需要 `{Selected...}` token 的动作无选中项时不可执行，只依赖 `{WorkspaceRoot}` 的动作不依赖选中项；成功/失败写入 Footer，不等待退出码，不自动刷新。

T03 和 T04 完成后一起做人工调试验收，覆盖默认 `OpenDiff`、`Open`、`ExploreTo`、`Edit`、`Commit` 以及一组手改配置。

## 6. 明确不做

T03-T08 为了保持配置驱动和实现轻量，明确不做以下内容：

- 不为 `OpenDiff`、`Open`、`ExploreTo`、`Edit`、`Commit` 写五套专用命令。
- 不根据 Git 状态类型做复杂可用性判断。
- 不检测 TortoiseGit 是否安装，不自动搜索安装路径。
- 不验证 TortoiseGitProc 参数是否真的完成 diff/commit；参数可由用户配置调整。
- 不等待外部进程退出，不读取退出码，不推断 commit 是否完成。
- 不做 Windows 之外的平台适配；当前默认动作按 Windows 工具链设计。
- 不做配置编辑 UI，只继续依赖全局 JSON 和 workspace `.HaloCreek/config.json`。

## 7. 验收标准

- 默认配置下，工具栏显示 `OpenDiff`、`ExploreTo`、`Edit` 在左侧，`Commit` 在右侧，`Refresh` 仍在最右侧。
- 默认配置下，`Open` 不显示按钮，但双击 Git 变更行会执行 `Open`。
- 修改配置后，按钮数量、按钮文本、左右排布、双击动作都能随配置变化。
- 需要 `{Selected...}` token 的动作在无选中项时不可执行；只依赖 `{WorkspaceRoot}` 的动作不依赖选中项。
- 动作执行成功只表示 HaloCreek 成功启动了配置的外部进程。
- 动作执行失败时 Footer 有明确失败信息。
- `GitViewModel` 不直接调用 `Process.Start`。
- UI 不包含 `OpenDiff` / `Commit` 等固定命令假设，除了默认配置提供这些动作。
