# HaloCreek 宣传文章草稿

## 标题

我做了一个给 Codex CLI 用的本地工作流壳层：HaloCreek

## 正文

最近一段时间，我在持续用 Codex CLI 做日常开发。用下来最大的感受是：Agent 写代码这件事本身已经越来越可用了，但真正影响效率的，往往不是“能不能生成代码”，而是生成代码前后那一圈人工介入操作。

比如：

- 这次要在哪个 workspace 里启动？
- 上一次 session 说到哪了，能不能快速恢复？
- 初始 prompt 写过一遍了，能不能拿回来改一下再跑？
- Agent 改了哪些文件，哪些已经看过，哪些又被后续修改覆盖了？
- 审阅时发现问题，怎么快速把反馈继续发给当前 session？
- 终端、Git 工具、历史记录、文件管理器之间来回切，能不能少一点？

这些问题单独看都不大，但在 human-in-the-loop 的 Agent 编程流程里，它们会反复出现。每一次切换窗口、复制路径、找历史、重新组织上下文，都在打断开发者的判断节奏。

这就是 HaloCreek 试图解决的问题。

### 定位：不是替代 IDE，而是本地工作流壳层

HaloCreek 不是一个新的 IDE，也不是云端 Agent 平台，更不是多 Agent 自动编排系统。它的定位很简单：一个面向 Windows + WSL + Codex CLI 的本地桌面工作流壳层。

Codex CLI 仍然负责真正的代码任务；Git、终端、TortoiseGit、编辑器也都还是原来的工具。HaloCreek 只做一件事：**把个人 Agent 编程过程中那些高频、低价值、容易打断心流的操作，收束到一个轻量桌面入口里。**

### 解决什么

目前它主要解决几个痛点。

#### 1. workspace 和 session 的启动摩擦

日常使用 Codex CLI 时，我经常需要在不同仓库之间切换，还要确认当前路径、启动新 session、恢复旧 session、把终端窗口重新拉到前台。HaloCreek 把 workspace 选择、prompt 编辑、Launch、Resume、ongoing session 管理放在一起。你可以直接在桌面应用里写 prompt，用快捷键启动 Codex，也可以从历史 session 里恢复或重用初始 prompt。

#### 2. 历史上下文的找回成本

Agent 编程不是一次性问答。很多时候，一个 session 里的初始目标、最近追问和最近回复都很有价值。HaloCreek 会读取本机 Codex 历史，并按当前 workspace 过滤，只展示和当前仓库相关的 session。这样想恢复一段上下文、复用一个 prompt、继续某个未完成思路时，不需要在文件系统或终端历史里翻来翻去。

#### 3. AI 修改后的人工审阅

这是我认为 Agent 编程里最关键、也最容易被低估的环节。Agent 可以快速改代码，但开发者仍然需要判断：这段改动是否符合意图？有没有越界？哪些文件已经看过？看过之后 Agent 又改了同一个文件怎么办？

HaloCreek 的 Review 面板围绕这个问题设计。它会展示当前 Git 修改，同时维护一个独立的 Reviewed 快照。文件被标记为 Reviewed 后，如果后续 working tree 又发生变化，它会重新进入 Unreviewed 状态。这样 **“我已经审过的内容”** 和 **“审过之后又出现的新变化”** 可以被区分出来。

这个设计不替代 Git staging area，也不改变真实 Git index。它只是为人工审阅提供一个轻量进度层，让开发者在 AI 持续修改的过程中更容易知道：**哪里还需要看，哪里已经确认过，哪里又被动过。**

#### 4. 审阅和追问之间的切换

真实审阅时，经常不是简单地看完再提交。你可能会问 Agent：“这里为什么这么写？”也可能会要求它“只改这一小块，不要动别的地方”。HaloCreek 在 Review 面板里保留了 prompt 入口，可以把审阅意见直接发送给当前前台 Codex session，也可以用当前审阅 prompt 拉起新 session。

#### 5. 保留本地工具链的可控性

我不希望这个工具变成一个接管一切的黑盒。HaloCreek 当前仍然依赖本地 WSL、tmux、Codex CLI、Windows Terminal、Git 和 TortoiseGit。Diff 还是外部 diff，Commit 还是外部 Git 工具，代码编辑也还是你自己的编辑器。

这意味着它不是最“全自动”的方案，但它更符合我对个人 Agent 编程工具的判断：自动化应该发生在低价值重复动作上，关键判断权应该继续留给开发者。

### 核心观点

所以 HaloCreek 的核心观点不是“让 Agent 全自动完成开发”，而是：

**让开发者更低摩擦地介入 Agent 编程流程。**

启动、恢复、审阅、追问、查看修改、回退到已审阅版本，这些动作越顺，开发者就越能把注意力放在真正重要的问题上：需求是否准确，设计是否合理，修改是否可信。

### 当前边界

当前 HaloCreek 还处在早期阶段，边界也很明确：

- 它优先面向 Windows + WSL + Codex CLI 的本地开发环境。
- 它不替代 IDE，不做语言服务或调试器。
- 它不做云端协作、账号系统或团队权限。
- 它不追求多 Agent 编排，也不把任务队列和自动流水线作为当前目标。
- 它更关注个人开发者在 human-in-the-loop 流程里的实际摩擦。

如果你也在用 Codex CLI 或类似的 coding agent，并且发现真正麻烦的不是“让 Agent 写代码”，而是写完之后如何审、如何追问、如何继续、如何不丢上下文，那么 HaloCreek 想解决的就是这个问题。

**它不是要把开发者从流程里拿掉，而是让开发者更轻松、更清楚、更可控地留在流程里。**
