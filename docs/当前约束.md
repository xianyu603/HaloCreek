# 当前约束

## Workspace 路径

当前 HaloCreek 在 Windows 端运行时，内部 workspace 路径只维护 Windows 路径。

具体来说，配置读取、目录选择、状态展示、Git 操作、workspace 级配置查找、历史记录过滤等应用内消费方，都以 Windows 可直接访问的路径作为 workspace 的唯一输入形态，例如：

```text
D:\work\HaloCreek
```

对于需要交给 WSL 执行的场景，例如启动 Codex，会在边界处把 Windows 路径转换成 WSL mount 路径：

```text
D:\work\HaloCreek
/mnt/d/work/HaloCreek
```

这个转换只发生在调用 WSL 命令之前，不改变 HaloCreek 内部保存和传递的 workspace 表示。

## 配置路径

配置中的 workspace 路径也只接受 Windows 路径，尤其是 `DefaultWorkspacePath`。

推荐写法：

```json
{
  "DefaultWorkspacePath": "D:\\work\\HaloCreek"
}
```

不支持把 WSL 原生路径或 WSL UNC 路径作为配置输入，例如：

```text
/home/ck/project
\\wsl.localhost\Ubuntu\home\ck\project
\\wsl$\Ubuntu\home\ck\project
```

这些路径即使在部分 Windows API 中可访问，也不属于当前 workspace 契约。当前实现不会保证它们在配置查找、历史记录匹配、Git 操作、终端启动等消费方之间保持一致。

## 历史记录匹配

Codex 历史记录里的 `cwd` 可能是 WSL 路径。当前支持的匹配边界是 Windows 盘符路径和对应 WSL mount 路径之间的等价关系，例如：

```text
D:\work\HaloCreek
/mnt/d/work/HaloCreek
```

不承诺匹配 WSL 原生 home 路径和 Windows WSL UNC 路径之间的等价关系，例如：

```text
/home/ck/project
\\wsl.localhost\Ubuntu\home\ck\project
```

如果未来要支持 WSL 原生 workspace，需要先扩展产品契约，再统一处理选择、配置、路径归一化、历史记录匹配、Git 操作和终端启动等链路。
