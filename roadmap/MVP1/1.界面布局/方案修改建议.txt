ViewModel 如何组织？ 不引入DI，但是需要有集中管理组件初始化，依赖关系注入的地方。暂时放在App.OnFrameworkInitializationCompleted()
设计期/占位数据怎么处理？在框架内部处理，要保证接口是正式的格式，需要显式隔离Mock实现和上层接口，方便未来替换。例:
  HistorySessionsViewModel
    -> SessionHistoryService
        -> 暂时返回 mock SessionInfo
  public sealed class ConfigService
  {
      public AppConfig LoadEffectiveConfig(WorkspaceInfo? workspace)
      {
          return AppConfig.DefaultForMvp1;
      }
  }
文件拖入能力放在哪里？应当走 独立 DragDropService ， 历史session和ongoing session很可能需要用到类似的DragDrop功能
历史 session 解析模块现在是否做格式防御？不止需要将逻辑限制在 SessionHistoryService 内，在SessionHistoryService内也要做接口隔离式的防御，如
  SessionHistoryService:
  - 面向业务
  - 决定“历史 session 列表需要哪些信息”
  - 做搜索、排序、过滤、映射、后续操作准备

  ICodexSessionReader / CodexSessionFileReader:
  - 面向 Codex 当前实现
  - 知道文件路径
  - 知道文件格式
  - 把不稳定的 Codex 原始格式转换成稳定的中间数据

ongoing session 是否现在接 tmux？ 配置系统是否现在实现？ Git tab 是否现在接真实 git？ 这三条需要从方案移除 scope不应当在方案中讨论
是否做响应式布局？本项目不做复杂响应式布局 保证简单缩放体验即可
